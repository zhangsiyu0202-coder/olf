# Overleaf 类网站应用 MVP - 需求规格说明书

## 1. 目标说明

本规格用于约束整个项目的第一阶段实现范围。目标不是一次性做出完整的 Overleaf，而是先做出一个可用、可扩展、能持续演进的在线 LaTeX 编辑网站应用。

第一阶段必须围绕以下核心闭环展开：

1. 用户创建项目
2. 用户管理项目文件
3. 用户在线编辑 `.tex` 等源文件
4. 用户触发编译并查看 PDF 预览
5. 用户关闭页面后再次进入，项目状态仍然存在

只要上述闭环没有稳定跑通，均不得优先实现高复杂度附加能力。

## 2. 工程原则

### 2.1 简单优先

- 遵循 KISS：优先单体架构或清晰边界的少量服务，不为未来假想规模提前拆散系统。
- 遵循 YAGNI：不预先实现双机房、GlusterFS、Redis 集群、多 AI Provider、复杂快照增量算法等能力。
- 同一能力先做一条稳定路径，再考虑抽象和扩展。

### 2.2 避免重复

- 文件元数据、项目权限、编译状态、AI 调用配置都必须有单一真相来源。
- 相同业务规则在第三次出现前必须抽象为共享模块或共享函数。

### 2.3 低耦合高内聚

- 编辑器、文件存储、编译、AI 助手必须通过明确接口通信。
- 每个模块只处理单一核心职责，避免“一个服务既管项目又管编译又管 AI”式的业务缠绕。

### 2.4 可读性优先

- 命名应直接反映业务含义。
- 控制函数长度和嵌套层级，优先使用提前返回。
- 先让主流程清晰，再考虑优化。

## 3. 第一阶段范围（P0）

### 3.1 必须实现

- 用户登录和基础会话管理
- 项目创建、重命名、删除、列表展示
- 项目内文件和目录的创建、读取、更新、删除、移动
- 基于浏览器的 LaTeX 文本编辑器
- 手动触发编译
- 编译日志展示
- PDF 预览
- 基础错误提示和行号定位
- 项目数据持久化

### 3.2 可以简化实现

- 权限模型先只支持 `owner` 和 `collaborator`
- 文件版本先只支持“最近一次可恢复快照”或显式保存版本
- 存储先使用 PostgreSQL 元数据 + 本地文件系统或 S3 兼容对象存储
- 编译先支持 `pdflatex`
- 协作先支持单用户或同用户多标签页一致性

### 3.3 第一阶段不做

- GlusterFS 双副本和跨机房高可用
- 100+ 并发用户压测目标
- 复杂增量快照算法
- 多 AI 提供商切换
- arXiv 集成
- xelatex / lualatex
- 自动扩缩容和 Kubernetes 生产编排
- 完整多人实时协作冲突解决

## 4. 第二阶段范围（P1）

- 多人实时协作编辑（基于 Yjs）
- 项目分享与邀请
- 更完整的版本历史
- 编译缓存
- AI 问答助手
- 编译错误智能诊断

## 5. 第三阶段范围（P2）

- 多节点编译集群
- 高可用存储
- arXiv 检索和导入
- 多模型 AI Provider 抽象
- 成本优化、上下文压缩、复杂观测与运维能力

## 6. 核心用户故事

### US-001 项目编辑闭环

作为用户，我希望在网页中创建一个 LaTeX 项目、编辑主文件、触发编译并看到 PDF 结果，这样我可以完成基础写作工作。

验收标准：

- [ ] 用户可以创建项目并进入编辑页
- [ ] 用户可以编辑 `main.tex`
- [ ] 用户可以点击编译按钮提交任务
- [ ] 系统返回编译状态、日志和 PDF 结果
- [ ] 刷新页面后仍能恢复项目内容

### US-002 文件树管理

作为用户，我希望像 Overleaf 一样管理项目中的多个文件和目录，这样我可以组织章节、图片和参考文献。

验收标准：

- [ ] 支持创建文件和目录
- [ ] 支持重命名、删除、移动
- [ ] 支持 `.tex`、`.bib`、图片等常见文件类型
- [ ] 文件树操作后界面和后端状态一致

### US-003 编译错误反馈

作为用户，我希望在编译失败时看到可读的错误信息和大致定位，这样我可以快速修复问题。

验收标准：

- [ ] 返回结构化编译结果
- [ ] 错误至少包含文件名、行号、消息
- [ ] 前端可以在编辑器中定位错误

## 7. 非功能要求

### 7.1 性能

- 项目页首屏可交互时间目标：< 3s
- 常规文本文件读取目标：< 200ms
- 单次编译请求提交目标：< 300ms
- 普通文档编译完成目标：P95 < 8s

### 7.2 可靠性

- 编译失败不影响编辑能力
- 后端单模块故障应返回明确错误，不得导致整站白屏
- 所有关键写操作必须有明确错误处理

### 7.3 可维护性

- 核心模块必须有边界清晰的目录结构
- API、数据模型、状态流转需要有最小必要文档
- P0 核心路径需要基础自动化测试

## 8. 与现有子规格的关系

### 8.1 file-storage

保留“项目文件管理”的核心目标，但第一阶段不采用 GlusterFS 双副本设计，先落到更简单的元数据 + 对象存储方案。

### 8.2 latex-compiler

保留“异步编译 + 日志 + 结果查询”的核心目标，但第一阶段只做单队列、单机或少量 worker 的稳定实现。

### 8.3 ai-module

AI 模块不是第一阶段核心闭环的一部分。只有在“编辑 + 编译 + 预览”链路稳定后，才进入正式开发。
