# AGENTS 协作规范

本文件用于约束本仓库内的人类开发者与 AI Agent 协作方式。所有新增内容必须遵守以下规则。

## 1. 总目标

本项目目标是实现一个模仿 Overleaf 的网站应用。当前阶段已经从“MVP 打通”进入“成品化阶段”。

当前阶段的核心目标是：

1. 项目创建与管理
2. 文件和目录管理
3. 在线 LaTeX 编辑
4. 异步编译
5. PDF 预览
6. 自动快照与版本恢复
7. AI 问答助手与编译错误诊断
8. 协作编辑基础能力
9. 协作文档状态持久化与在线状态
10. 项目成员与邀请系统
11. 正式登录体系与可信 Cookie 会话
12. 个人空间 / 组织空间 / 团队空间
13. PostgreSQL 元数据与对象存储适配
14. 审计日志与版本事件时间线
15. 基于 `fileId` 的稳定文件主键
16. 评论与批注系统
17. `latexmk` 多轮编译与编译缓存

说明：

- 已经打通的主链路不能被后续扩展破坏。
- 新增模块必须围绕“稳定性、可恢复性、可扩展性”服务，而不是只追求功能堆叠。
- 当前阶段允许对存储层和服务边界做升级，但必须有明确迁移路径，不能推翻现有模块边界。

## 2. 当前阶段优先级

按照当前阶段约束，后续工作默认按以下优先级推进：

1. 版本回放与差异比较
2. 评论锚点和 PDF 预览联动
3. 更细粒度权限矩阵
4. 更复杂的分布式与高可用能力

禁止事项：

- 不允许为了引入 AI 或协作而回退已经稳定的编译链路。
- 不允许在快照机制尚未稳定前，直接实现复杂增量版本算法。
- 不允许在只有单一 AI Provider 时，先做过度抽象的多 Provider 平台。

## 3. 工程原则

### 3.1 简单优先

- 遵循 KISS：优先简单直接的实现。
- 遵循 YAGNI：不为未来假想需求提前堆砌抽象。
- 新增模块前，先确认现有模块是否已经足够承载需求。

### 3.2 避免重复

- 业务规则、错误码、共享 DTO 必须有单一真相来源。
- 同样逻辑出现超过两次时，必须抽象到共享层。

### 3.3 低耦合高内聚

- `apps/web` 只处理 UI 与交互。
- `apps/api` 只处理接口聚合、鉴权、项目和文件业务。
- `workers/compiler` 只处理编译任务。
- `packages/runtime-store` 只处理当前阶段的存储抽象与仓储适配，不直接承载 HTTP 或 UI 逻辑。
- `packages/ai-assistant` 只处理 AI 问答上下文拼装、Provider 调用与本地兜底策略。
- `packages/contracts` 只处理共享接口、DTO、错误码。
- `packages/shared` 只处理纯工具和无业务副作用的共享能力。

### 3.4 可读性优先

- 命名必须直接表达业务意图。
- 控制函数长度和嵌套深度，优先使用提前返回。
- 注释用于解释“为什么”，不是重复“代码做了什么”。

### 3.5 童子军规则

- 修改现有代码时，顺手修正低风险的命名、注释和结构问题。
- 不得把明显的脏代码继续扩散到新模块。

## 4. 成品化阶段设计约束

### 4.1 自动快照

- 自动快照优先采用简单可靠的全项目快照方案。
- 第一版默认允许“双触发”：
  - 成功编译后创建里程碑快照
  - 编辑达到时间阈值后创建检查点快照
- 第一版恢复粒度优先支持整项目恢复，不强求单文件恢复。

### 4.2 存储层升级

- 允许从当前文件化持久层逐步升级到 PostgreSQL + 对象存储。
- 升级必须优先收敛到 `packages/runtime-store`，不能把存储细节扩散进 `apps/api` 或 `workers/*`。
- 元数据与快照索引必须有单一真相来源。

### 4.3 AI 问答助手

- 第一版 AI 只做问答助手，不默认实现自动补全、自动改写和多 Provider 抽象。
- AI 默认上下文来源应收敛到：
  - 当前文件内容
  - 当前选中文本
  - 最近一次编译日志
- AI 生成结果必须由用户确认后再写回编辑器。

### 4.4 协作编辑

- 当前协作实现默认基于 `Yjs + WebSocket + CodeMirror 6`。
- 协作房间现在必须围绕 `projectId + fileId` 建模，路径只用于展示和回放。
- 协作文档状态持久化必须继续收敛在 `packages/runtime-store`，不能让前端或 API 直接处理底层状态文件。
- 评论、团队权限和审计日志应作为协作上层能力追加，不得反向侵入编辑器内核。

### 4.5 项目成员与邀请

- 项目成员关系、邀请链接和访问控制属于项目层能力，不属于文件协作内核。
- “谁能进入同一个论文项目”必须通过成员关系或邀请机制解决，不能把协作房间 key 暴露成产品入口。
- 项目显式成员、工作空间成员和邀请链接必须共同构成访问控制边界，不能各自为政。
- 当前阶段默认以正式登录体系和可信 Cookie 会话为主，不再以本地演示用户会话作为主路径。

### 4.6 编译体验增强

- 项目级编译设置应以 `rootFile + compileEngine` 作为单一真相来源，统一收敛到项目元数据。
- 编译任务入队时必须快照当前编译设置，避免排队期间设置变化导致结果漂移。
- 前端编译日志面板应优先展示结构化诊断和跳转定位能力，而不是只堆原始日志文本。
- 宿主机编译应优先使用 `latexmk`，不可用时再回退到多次引擎执行。
- 编译缓存必须基于项目内容哈希、根文件和引擎共同决定，避免错误复用旧产物。

### 4.7 平台化能力

- 组织、团队、审计日志和版本事件默认依赖 PostgreSQL。
- 项目文件、快照归档、协作状态和编译产物默认通过 `packages/runtime-store` 的 blob 门面读写，允许切换到 S3/MinIO。
- 编译 Worker 允许在 `host` 与 `docker` 两种执行模式之间切换，但不得改变任务协议和状态机。

## 5. 文件与注释规范

所有代码文件必须同步维护注释，并满足以下要求：

1. 文件顶部提供文件级块注释，说明职责、数据流、依赖和扩展点。
2. 重要函数提供函数级注释，解释做什么、为什么、影响是什么。
3. 复杂逻辑块提供必要的块注释。
4. 文件尾部保留或更新 `Code Review` 注释块，记录设计权衡、边界条件和潜在风险。

说明：

- 本规范适用于 `C++`、`Go`、`TypeScript`、`QML` 及其他会承载业务逻辑的源码文件。
- 配置文件、JSON、YAML、Markdown 文档不要求追加 `Code Review` 注释块。

## 6. 模块边界规则

- 不允许在 `apps/web` 中直接复制后端业务规则。
- 不允许在 `workers/compiler` 中写项目管理逻辑。
- 不允许把数据库实体定义散落到多个模块重复维护。
- 不允许在 `apps/api` 与 `workers/compiler` 中各自复制一套项目或任务持久化逻辑。
- 不允许把快照逻辑散落到编译模块、编辑模块和 AI 模块各自维护。
- 不允许把 AI prompt 组装逻辑同时写在前端和后端两边。
- 共享类型先放 `packages/contracts`，共享工具先放 `packages/shared`。

## 7. 规格优先级

当实现与规格冲突时，按以下顺序判断：

1. 当前 `AGENTS.md` 的阶段性约束
2. 当前阶段新增的设计/路线图文档
3. `.kiro/specs/overleaf-mvp/` 中仍然有效的基础架构约束
4. 具体子模块规格
5. 局部实现细节

如果旧规格与当前阶段目标冲突，应优先按当前阶段约束修订旧规格，而不是继续沿用过期约束。
