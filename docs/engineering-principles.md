# 工程原则

本项目的所有实现都必须围绕以下原则推进。

当前阶段说明：

- 项目已经完成 MVP 主链路打通。
- 后续扩展以“可靠成品化”为目标，而不是继续停留在原型阶段。
- 自动快照、存储升级和 AI 问答助手均属于当前阶段允许推进的事项。

## 1. KISS

系统复杂度是可维护性的主要敌人。只要能满足当前业务需求，就优先使用最直接、最容易验证的实现。

约束：

- 不提前引入多节点、主从、集群等高复杂度方案
- 不提前设计过深的抽象层
- 不为了“以后可能有用”而增加当前认知负担

## 2. YAGNI

只解决当前已经被规格确认的问题，不为假想需求编写复杂代码。

约束：

- 当前阶段不提前实现完整多人实时协作
- 当前阶段不提前实现多 AI Provider 抽象
- 当前阶段不提前实现高可用存储集群
- 当前阶段不提前实现复杂增量快照算法

## 3. DRY

任何业务规则、配置和错误码都必须只有一个权威来源。

约束：

- 共享 DTO 放在 `packages/contracts`
- 共享纯工具放在 `packages/shared`
- 不在前后端复制同一套校验逻辑说明文字和魔法数字

## 4. 高内聚低耦合

每个模块只承担单一核心职责，模块之间通过接口协作。

约束：

- 前端负责界面与交互
- API 负责业务编排与访问控制
- 编译 Worker 负责异步编译
- 存储层负责持久化，不侵入 UI 和任务流控制

## 5. 可读性优先

代码首先是给人维护的，其次才是给机器执行的。

约束：

- 命名即文档
- 优先提前返回，减少深层嵌套
- 复杂代码必须解释为什么这么写

## 6. 童子军规则

每次改动都应该让局部代码比之前更干净一点。

约束：

- 修复明显误导性的命名
- 清理过时注释
- 消除低风险重复代码
