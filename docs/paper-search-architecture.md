# 论文多源检索设计

本文件用于定义论文模块的正确产品方向，避免把当前已接入的数据源误写成唯一主通道。

## 1. 核心结论

论文模块的目标不是围绕某一个论文源构建，而是：

1. 多源聚合检索
2. 分层全文获取
3. 统一项目文献资产落库

这里的重点是：

- 多个论文源的价值在于增加搜索广度、学科覆盖和召回质量
- 全文获取是另一层能力，不等于“某个源能搜索”就等于“默认能拿全文”

## 2. 正确的分层

### 2.1 搜索发现层

职责：

- 接收用户查询
- 调用多个论文源
- 返回统一结果结构
- 做来源标记、去重、排序和结果合并

候选来源：

- `arXiv`
- `Semantic Scholar`
- `PubMed`
- `Crossref`
- `OpenAlex`

说明：

- 这一层的核心目标是“搜得更全”，不是“直接拿全文”
- 不同来源必须保留清晰标签，不能混成单一模糊结果池

### 2.2 全文获取层

职责：

- 根据论文结果决定是否可读
- 获取 PDF、HTML 或可解析文本
- 供前端阅读器、摘录、总结与回流使用

候选来源：

- `arXiv PDF`
- `PubMed Central`
- 开放 PDF 链接
- 出版社站点公开页面
- 用户上传 PDF

说明：

- 这一层的核心目标是“读得进去”
- 并不是所有搜索结果都一定能进入全文阅读

### 2.3 项目文献资产层

职责：

- 把多源论文统一落到项目文献库
- 管理 BibTeX、引用键、来源、元数据和阅读状态
- 为写作区提供单一真相来源

最小统一字段建议：

- `source`
- `sourceLabel`
- `sourceId`
- `title`
- `authors`
- `published`
- `summary`
- `doi`
- `venue`
- `abstractUrl`
- `pdfUrl`
- `fullTextAvailable`
- `importedAt`

## 3. 各论文源的定位

### 3.1 arXiv

强项：

- 开放获取友好
- PDF 获取简单
- `AI / CS / Math / Physics` 覆盖强

在平台中的位置：

- 既是搜索发现源
- 也是当前最容易打通的全文获取源

当前状态：

- 已落地

### 3.2 Semantic Scholar

强项：

- 通用学术发现
- 作者、引用、相关推荐等元数据能力更强
- 能明显补足不在 `arXiv` 的论文线索

在平台中的位置：

- 优先作为搜索发现源
- 不是默认全文源

当前状态：

- 尚未接入当前仓库实现

### 3.3 PubMed

强项：

- 医学、生物、药学方向覆盖强
- 能补足生命科学文献

在平台中的位置：

- 搜索发现源
- 对接 `PMC` 或外部全文链接时可作为全文入口线索源

当前状态：

- 尚未接入当前仓库实现

## 4. 当前仓库的实现边界

当前已实现的是：

- `packages/paper-assistant` 内的 `arXiv` 检索、阅读、BibTeX 和研究 Agent
- API 层论文搜索、导入、阅读、PDF 缓存和摘录接口
- 前端论文检索与论文阅读面板

当前尚未实现的是：

- 多源聚合搜索
- 多源结果去重与合并
- `Semantic Scholar / PubMed` 源适配
- 分来源的全文可读性判定

因此，当前正确表述应当是：

- 论文模块已经落地第一条数据源链路：`arXiv`
- 论文模块的目标架构是：多源聚合检索 + 分层全文获取

而不应写成：

- `arXiv` 是唯一主通道

## 5. 对前端与 API 的约束

### 5.1 前端

- 搜索结果必须显示明确来源标签，例如：
  - `arXiv`
  - `Semantic Scholar`
  - `PubMed`
- 不同来源结果可以聚合展示，但不能去掉来源标识
- 阅读页应区分：
  - 摘要可读
  - PDF 可读
  - 全文不可直接获取

### 5.2 API

- API 返回值必须保留来源字段
- 统一搜索接口应返回分组或可区分的聚合结果
- 导入项目文献库时，必须保留原始来源和原始 ID

## 6. 默认扩展顺序

建议顺序：

1. 保持当前 `arXiv` 链路稳定
2. 接入 `Semantic Scholar` 作为通用学术发现源
3. 接入 `PubMed` 作为医学/生物补充源
4. 接入 `Crossref` 做 DOI / venue 元数据补全
5. 补强统一去重、排序、缓存与限流

## 7. 决策原则

- 不把某一个论文源写成产品唯一中心
- 不把“搜索发现”与“全文获取”混成同一层
- 不让前端直接感知某个 LangChain 组件细节
- 不让项目文献库丢失来源信息

一句话总结：

论文模块应当被实现为“多源聚合搜索器 + 分层全文获取器 + 项目文献资产入口”，而不是某一个论文网站的单源包装。
